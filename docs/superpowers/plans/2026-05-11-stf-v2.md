# STF v2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the v2 redesign of the `spec-tests-first` plugin — six-phase cycle (`spec → build → review → fix → validate → ship`), per-AC red-green-refactor inside `/spec-tests-first:build`, new `/spec-tests-first:review` and `/spec-tests-first:fix` phases, stack-aware test-layout profiles with monorepo support, inline `/spec-tests-first:ship` with zero external dependencies.

**Architecture:** The plugin is composed of Markdown SKILL.md files (per phase), Markdown agent files (with YAML frontmatter), and one slash command. The "implementation" is therefore authoring/editing those Markdown contracts and verifying behavior end-to-end on a sample feature. Each slice produces a self-contained set of file changes followed by frontmatter validation, cross-reference checks, and (where useful) a smoke run on a throwaway sample feature.

**Tech Stack:** Claude Code plugin format (SKILL.md with YAML frontmatter, agent .md, slash commands). Bash + `gh` + `git` for cycle integration. No runtime test framework on the plugin itself — verification is (a) YAML frontmatter parses cleanly, (b) cross-references between phases resolve, (c) a smoke run on a throwaway sample feature exits cleanly.

**Source of truth for prose:** `docs/superpowers/specs/2026-05-11-stf-v2-design.md`. Each task points at the relevant spec section rather than restating it.

## Slice 3 — Extend `code-reporter` with finding-id + Status column

Goal: the reporter emits a stable `finding-id` per finding (`SEC-001`, `QUA-014`) and initializes a `Status: pending` field on each. `/spec-tests-first:fix` mutates the status later.

### Task 3.1: Add "Finding IDs and Status column" section to the agent prompt

**Files:**
- Modify: `plugins/spec-tests-first/agents/code-reporter.md`

- [ ] **Step 1: Read the current agent prompt to find the right insertion point**

The new section must come *after* the existing aggregation rules ("# Aggregation" or similar) and *before* the output-template section. Locate that boundary.

- [ ] **Step 2: Insert the new section**

Insert this block at the boundary identified in Step 1:

````markdown
## Finding IDs and Status column

For every finding you aggregate, prepend a stable ID:
- Security findings → `SEC-001`, `SEC-002`, ... (sequential, in the order the security reviewer listed them)
- Code-quality findings → `QUA-001`, `QUA-002`, ... (same)

IDs are stable across runs only within the same report. A fresh `/spec-tests-first:review` produces fresh IDs — never carry IDs forward from prior reports.

For each finding row, emit a `Status` line on its own line, initialized to `pending`:

```
**[SEC-001] [Critical] auth/login.py:47** — SQL injection via String concatenation in user lookup
Why: ...
Suggested fix: ...
Status: pending
```

The `Status` line is parser-stable — `/spec-tests-first:fix` greps for it and mutates the value to `fixed` / `skipped` / `deferred` in place.

Never alter status values you find in an existing report — that's `/spec-tests-first:fix`'s job. If you're producing a fresh report (you always are; the reporter is invoked per-`/spec-tests-first:review`-run and writes a new timestamped file), all `Status` fields start at `pending`.

The reporter never preserves a `## Fix log` section from a prior report. Each report is a fresh starting point; the fix log is created and appended only by `/spec-tests-first:fix`.
````

### Task 6.7: Write Step 6 — Per-AC RGR loop (the heart of the rewrite)

- [ ] **Step 1: Write the loop structure**

For each AC-ID in dependency order (within a service for monorepos), execute the 4-stage cycle:

**RED**
1. Write ONE test for the AC in the right file (per profile / per service). Test name embeds the AC-ID (e.g., `test_AC_1_1_*`). For error-path ACs, use the discriminating signal from spec Section 3.
2. Dispatch `test-runner` subagent: `test_command` (service-specific in monorepos), `test_path` (the test file for this AC), `expected_ac_ids = ["AC-N.M"]` (single-element list).
3. Parse the returned JSON (extract first `````json` fenced block, parse defensively).
4. Expected: AC in `failed` array.
   - AC in `passed` → test is too permissive. Tighten assertion, retry RED. Show the user the test and the tightening intent.
   - AC in `errored` → fix the test (likely syntax/import), retry.
   - `runner_failed` / `json_parse_failure` → surface `stdout_tail`, stop.

**GREEN**
1. Read `codebase-map.md` for existing modules — don't duplicate.
2. Write the minimal production code to pass the test. Honor the discriminating signal verbatim.
3. Dispatch test-runner with the same inputs as RED. Parse JSON.
4. Expected: AC in `passed`.
   - AC in `failed` → analyze message, fix code (NOT test), retry. **Cap = 3 attempts per AC.**
   - Same failure twice in a row with no change in message → STOP early.
   - Cap hit → mark AC `fail` in spec-status.md, surface a one-paragraph summary of all 3 attempts + the unresolved failure, ask via AskUserQuestion: (a) keep the branch for debugging, (b) revert this AC's changes and continue with next, (c) stop the whole build.

**REFACTOR** (optional)
1. Clean up only if helpful (remove duplication, improve names, extract helpers). No new behavior.
2. Re-run test-runner. If green → next stage. If not → REFACTOR introduced a regression, revert it.

**REGRESSION CHECK**
1. Dispatch test-runner against `tests_root` for this service (or `tests/<feature>/` for single-service), with `expected_ac_ids` = all ACs implemented so far (including this one).
2. Parse JSON.
3. All previously-pass ACs still pass → mark this AC `pass` in spec-status.md. Next AC.
4. Regression → this AC's implementation broke something. Revert just this AC's diff (`git checkout` the files this AC touched; the previous ACs' files are preserved). Mark AC `blocked` with the regression note. Ask: continue with next AC or stop?

- [ ] **Step 2: Write the checkpoint logic**

Between AC iterations:
- For specs with ≤3 ACs → silent, no prompt.
- For specs with >3 ACs → pause at **capability cluster boundaries** (between `AC-1.*` and `AC-2.*`). AskUserQuestion: continue / inspect / stop.
